*goroutine are a lightweight application-level thread that has a separate independent execution.
*The main function in the main package is the main go-routine.
All goroutines are started from the main goroutine, these goroutines can then 
start multiple other goroutines, and so on.
But the main goroutine represents the main program, once it exits then it means that the entire program has exited.

* goroutines do not have parents or children,
- when you start a goroutine, it just executes alongside all other goroutines.
- Each goroutine exits only when its function returns,
The only exception to that is that:
"All goroutines exit when the main goroutine, the one that runs function main, exits."

Go schedular:
- golang schedular is a cooperative schedular, meaning non-preemptive, that there is no time-based preemption that is happening from the OS, 
So its a style of scheduling in which the OS never interrupts a running process to initiate a context switch from one process to another.
In fact the processes must voluntarily yield control periodically or when logically blocked on a resource.
There are some specific checkpoints where goroutines can yield its execution to other go-routine, these are called context switches.
For example, the runtime calls the scheduler on function calls to decide whether a new goroutine needs to be scheduled.
So basiclly, when a goroutine makes any function call, in that case the schedular will be called and a context switch might happen, meaning a new goroutine might be scheduled and its also 
possible that the existing goroutine continues its execution and no context switching happens.

Some example of context switching are:
-function calls, 
-garbage collection, 
-network calls, 
-channel operations,
-on using go keyword.
Now its just that the schedular gets the oppurtunity over here, it is upto the schedular whether to do a context switch or not.

Difference between goroutine and threads:
-goroutines are cheaper when compared to threads, they are only a few KBs in size and the stack can grow and shrink according to the needs of the application,
Whereas for a thread, the stack size has to be specified and is fixed and Operating system threads generally start with 1 MB of stack space, 
Since goroutines are cheap, you can launch hundread and thousands of goroutines while you can only launch a few thousand threads.

-** goroutines are multiplexed to a fewer number of OS threads, there might be only one thread in a program with thousands of goroutines.

-** The scheduling of goroutines is done by go runtime 
Whereas in case of Threads the scheduling of threads is done by the OS(Operating system) runtime, and therefore the context switching time of go-routines is much faster
than the context switching time of threads.

- goroutines communicate using channels, channels by design prevent race conditions from happening when accessing share memory using goroutines, 
channels can be thought of as a pipe using which goroutines communicate.

wait groups:
*The problem with go-routines was the main go-routine terminating before the 
go-routine completed or even began their execution.
*To wait for multiple go-rotuines to finish, we can use a wait group.
*A wait group is a synchronization primitive that allows go-routines to wait for each other.                     
*This package acts like a counter that blocks execution in a structured way until its internal counter becomes 0.
*wait group syntax: var <variable-name> sync.WaitGroup <--- this "sync.WaitGroup" is the datatype.
*import package "sync" to use this.
 
***wait groups have 3 vvimp methods :
No.1: wg.Add(int) //this indicates the no. of go routines to wait for, the int in the func acts like a counter. 
No.2: wg.Wait() //This method blocks the execution of code until the internal counter reduces to value=0.
No.3: wg.Done() //This method decreases the internal count parameter in Add() method by 1.

Channels:

* Channels are a means through which different go-routines communicates.
* Unbuffered channel: A channel that needs a receiver as soon as a message is emitted to the channel & Unbuffered channel does not hold any data.
* Buffered channel : Have capacity to hold data, On a buffered channel sending to a channel, blocks the go-routine, only if the buffer is full.
  - Receiving from a channel blocks only when the channel is empty.

  syntax of buffer channel: c:=make(chan <data_type>, capacity)
  example: c:=make(chan int, 5)  <-- this channel is meant to send and receive  integer data only and has a buffer of 10 values. 
The second argument (5) is the capacity of the channel. That means the channel can hold up to 5 integers without requiring a receiver.
The sender is allowed to keep sending until the buffer is full. Say you only inserted 2 values into a buffer of size 5, no blocking occurs And the moment you exceed buffer/capacity a deadlock will occur.
  
<------More on Buffered Vs Unbuffered Channels:------>
Unbuffered channel (make(chan int))

An unbuffered channel has capacity 0.

Every send (ch <- value) must have a corresponding receiver ready.
The sender and receiver synchronize at the same instant (this is called a rendezvous).
ch := make(chan int)

go func() {
    fmt.Println(<-ch)
}()

ch <- 100 // Works because receiver is waiting

Without a receiver:

ch := make(chan int)
ch <- 100 // Deadlock

because there's nowhere for the value to be stored.

Buffered channel (make(chan int, 5))

A buffered channel has storage for 5 values.

ch := make(chan int, 5)

ch <- 1
ch <- 2
ch <- 3

These sends succeed even though no receiver exists yet.

However, once the buffer is full:

ch := make(chan int, 2)

ch <- 1
ch <- 2
ch <- 3 // Blocks here until someone receives

The third send blocks because the buffer has no free space.

So a receiver is eventually needed if you want to keep sending or if you want to consume the values. It's just not needed immediately.

An easy analogy

Think of a mailbox.

Buffered channel
Capacity = 3

Mailbox:
+----+----+----+
|100 |200 |300 |
+----+----+----+

The sender can keep dropping letters into the mailbox until it's full.

If the mailbox is full, the sender must wait until someone removes a letter.

Unbuffered channel

There is no mailbox.

The sender must hand the letter directly to the receiver.

If nobody is there to take it, the sender waits forever.

Your statement, refined

For a buffered channel, we may not need a receiver immediately because values can be stored in the buffer. But once the buffer becomes full, any further send blocks until a receiver reads from the channel. For an unbuffered channel, every send requires a receiver to be ready at the same time; otherwise the program blocks and can deadlock.

Concurrency practices Cleaning up Go routines:
* whenever you launch a go-routine function, you must make sure that it will eventually exit.
* A go-routine that would never terminate , forever occupies the memory it has reserved. This kind of memory leak is called go-routine leak.
*go routines leak if they end up either blocked forever on I/O like channel communication or fall into infinite loops.

When to use buffered channel ?
Buffered channel are useful when you know how many go-routines you have launched, want to limit the number of 
go-routines you will launch, or want to limit the amount of work that is queued up.

Time out code:
One of the very famous Concurrency practices is to use "time-out-code"
* Most interactive programs must return a response within a certain amount of time.
And one of the thing that we can do with Concurrency in golang is to manage how much time a request or a part of a request has to run, so
Blocking timeout in select statement can be achieved by using After function of time package.
Signature of function: func After(d Duration) <- chan Time 
* The After function waits for Duration d to finish and then it returns the current time on a channel.
